## Démarrage Rapide

> 🔍**Vous avez recherché des compétences en codage IA, des compétences Claude Code, des compétences Curseur, des compétences Codex CLI, des compétences Gemini CLI, des compétences antigravité ou des bibliothèques `SKILL.md` installables ?**
>**Vous êtes au bon endroit.**👇### 1️⃣ What is this?

Awesome Omni Skills est un**catalogue de compétences et un environnement d'exécution installables**pour les assistants de codage IA. À la base, il s'agit d'un référentiel public de playbooks « SKILL.md » réutilisables, d'un référentiel organisé de dérivés de meilleures pratiques améliorés et des surfaces d'exécution CLI, API, MCP et A2A qui distribuent et exploitent le même catalogue.

Les apports autochtones saisissent les « compétences » de deux manières : les PR des contributeurs directs et les PR d'importation externe basés sur un référentiel examinés. Les dérivés sélectionnés uniquement en anglais sont ensuite proposés séparément dans « skills_omni/ » par le pipeline d'amélioration privé.

| 🧠**Compétences**| Playbooks réutilisables basés sur « SKILL.md » pour les assistants d'IA, y compris l'apport natif des PR contributeurs directs et les PR d'importation externe basés sur un référentiel examinés |
| 📦**Manifestes**| Manifestes, bundles et archives JSON générés |

## 🧩 Core Concepts

Avant de comparer les offres groupées ou de choisir un chemin d'installation, il est utile de comprendre ces cinq éléments de base :

| Concepts | Qu'est-ce que cela signifie |
|:--------|:-------------|
| 🧠**Compétences**| Playbooks `SKILL.md` réutilisables qui apprennent à un assistant comment bien exécuter un flux de travail |
| 📦**Catalogue Artefacts**| Génération de sorties JSON et d'archives permettant la recherche, la comparaison, le téléchargement et l'installation |
| 🔌**Configuration MCP**| Configuration côté client pour que les assistants découvrent Awesome Omni Skills via les outils MCP |
| 🤖**Exécution A2A**| Orchestration d'agent à agent pour la découverte, la recommandation et le transfert du plan d'installation |
| ✨**Sortie organisée**| `skills_omni/` — la surface améliorée maintenue par Awesome Omni Skills, distincte de l'apport natif en amont |

>**📝 Politique native/conservée :**
> - `skills/` accepte les apports natifs en amont dans n'importe quelle langue, qu'ils soient soumis directement ou proposés par le runtime privé de synchronisation externe
> - `skills_omni/` est toujours organisé et publié en anglais
> - `skills_omni/` est une surface à sens unique et ne revient pas à l'admission native
> - Les PR des contributeurs directs et les apports externes basés sur le référentiel partagent le même validateur, l'actualisation des métadonnées et le même chemin d'amélioration---

## 💡 Why Awesome Omni Skills

>**Pas seulement « un autre référentiel avec des compétences dans des dossiers ».**
> Awesome Omni Skills a un contrat plus solide et une surface d'exécution plus large.

| Si tu veux… | 📁 Référentiel de compétences typique | ✨ Superbes compétences Omni |
|:-------------|:------------|:--------------|
| Installer dans un véritable assistant | Copie manuelle ou script personnalisé | `npx Awesome-omni-skills`, installation guidée, interface utilisateur visuelle, `--skill` et `--bundle` sélectifs |
| Rechercher et comparer les compétences | Parcourir manuellement les démarques | Catalogue généré, filtrage, planification groupée, recherche, comparaison et recommandation |
| Utiliser les mêmes données dans tous les outils | Logique séparée par outil | Manifestes et catalogue partagés pour CLI, API, MCP et A2A |
| Configurer les clients MCP | Fichiers édités à la main | `config-mcp`, aperçus side-car locaux, recettes générées et écritures sur liste autorisée |
| Contenu du référentiel de confiance | Pelluchage minimal ou ponctuel | La porte de sécurité statique critique bloque les modèles dangereux tels que le contenu distant transmis au shell ou les instructions qui tentent d'exposer des invites, des secrets ou un contexte caché |
| Libérations de confiance | Emballage de qualité | Sommes de contrôle, archives signées, CI de version vérifiée par ClamAV + VirusTotal et publication en amont |
| Organiser l'admission de la communauté | Quelles que soient les terres, elles restent telles quelles | Apport natif dans `skills/`, dérivés anglais organisés dans `skills_omni/` avec attribution |---

## 🔌 Runtime Surfaces

Awesome Omni Skills n'est pas seulement une bibliothèque de compétences. Il expose**quatre surfaces d'exécution**qui consomment le même catalogue généré :

## 🛡️ Security and Release Posture

> Awesome Omni Skills propose une histoire de version et de vérification plus solide qu'un simple référentiel de démarques.### 🧪 Validation and Smoke Checks

```bash
npm run validate         # Skill validation and metadata generation
npm run build            # Full build pipeline
npm test                 # Legacy integration tests + Vitest unit suite
npm run test:unit        # Fast Vitest unit tests only (~500ms)
npm run test:coverage    # Vitest with V8 coverage
npm run smoke            # Full release preflight
```

<détails>
<summary>📋 <strong>Ce que le pipeline valide</strong></summary>

- ✅ Validation des compétences et génération de métadonnées
- ✅ Contrôle de sécurité critique sur l'admission native avant la fusion des relations publiques
- ✅ Outils de normalisation et de recatégorisation de taxonomie
- ✅ Génération de catalogue et d'archives
- ✅ Tests d'intégration hérités (assertions Python PTY + Node TUI)
- ✅ Suite d'unités Vitest pour la logique de notation, de recherche et de filtrage du catalogue
- ✅ Chemins de démarrage API, MCP et A2A
- ✅ Vérification des archives
- ✅ Contrat OpenAPI 3.1 servi via l'interface utilisateur Swagger dans `/docs`
- ✅ Contrôle en amont du package avec `npm pack --dry-run`</details>

<détails>
<summary>🔐 <strong>Posture de relâchement</strong></summary>

| Contrôle | Descriptif |
|:--------|:---------------|
| 🔒 Sommes de contrôle SHA-256 | Manifestes de somme de contrôle pour toutes les archives |
| ✍️ Objets signés | Signatures détachées sur les artefacts de version |
| 🤖 CI-appliqué | Vérification de la version dans CI avant publication |
| 🦠Portes scanner | ClamAV et VirusTotal sont tous deux requis lors de la vérification de la version avant la publication |
| 📦 Sortie GitHub | Génération automatisée des versions GitHub |
| 📋publication npm | À partir d'une archive tar vérifiée uniquement |
| 🔄 Libération automatique | Lors de la qualification, la compétence fusionne avec « principale » |

**La publication automatique se déclenche uniquement lorsqu'une fusion est modifiée :**
- `compétences/*/**`
- `skills_omni/*/**`
- `data/bundles.json`

Les modifications apportées uniquement aux documents**ne déclenchent pas**la publication du package.</details>

<détails>
<summary>🛡️ <strong>Contrat de sécurité du référentiel</strong></summary>

- L'admission autochtone n'accepte pas les compétences présentant des résultats de sécurité critiques.
- Le validateur statique bloque les modèles à haut risque avant qu'une compétence puisse fusionner dans le catalogue public.
- Les bloqueurs durs actuels incluent :
  - contenu distant transmis directement à l'exécution du shell
  - des instructions qui tentent de révéler des invites, des secrets ou un contexte d'exécution caché
- Les versions publiées passent par une deuxième couche plus stricte :
  - Numérisation ClamAV
  - Recherche de hachage VirusTotal
  - vérification de la somme de contrôle des archives
  - vérification de signature détachée dans la version CI

Il s’agit d’un avantage pour l’utilisateur, et pas seulement pour la plomberie : le catalogue privilégie les compétences installables qui ont déjà passé les mêmes barrières de sécurité restrictives que celles utilisées en interne par le projet.</details>

## 🗂️ Repository Layout

| Chemin | Objectif |
|:-----|:--------|
| 📂 `compétences/` | Compétences de création canonique et apport natif provenant de PR directs ou d'importations révisées basées sur un référentiel |
| ✨ `skills_omni/` | Dérivés améliorés gérés par Omni |
| 📖 `docs/` | Documentation des utilisateurs, des contributeurs, de l'architecture, des opérations et des spécifications |
| 📦 `dist/` | Manifestes, bundles, catalogue et archives générés |
| 📁 `données/` | Définitions de bundle et données de support statiques |
| 🧠 `packages/catalog-core/` | Runtime de catalogue partagé avec injection de dépendances `ICatalogStorageAdapter` |
| 🌐 `packages/serveur-api/` | API HTTP en lecture seule avec interface utilisateur OpenAPI/Swagger sur `/docs` |
| 🔌 `packages/serveur-mcp/` | Serveur MCP et side-car local |
| 🤖 `packages/serveur-a2a/` | Exécution A2A et orchestration des tâches |
| 🖥️ `packages/cli/` | Points d'entrée CLI unifiés, logique d'installation et TUI visuel Ink (ESM natif) |
| ⚙️ `outils/scripts/` | Scripts de validation, de génération, de publication et de test |
| 🧪 `vitest.workspace.js` | Configuration de l'espace de travail monorepo Vitest |

>**ℹ️**`dist/` est intentionnellement versionné car les artefacts générés font partie du contrat d'installation, de l'API, du MCP, de l'A2A, de Smoke et de la version.---

## Contribuer

Awesome Omni Skills accepte l'acquisition de compétences natives en amont sous « compétences/ ».

| Règle | Détails |
|:-----|:--------|
| 📥 Apport natif | Peut être approximatif, rédigé dans n'importe quelle langue et peut provenir de PR directs ou de succursales d'importation externes examinées |
| ✨ Sortie organisée | `skills_omni/` réservé aux dérivés Omni créés par l'automatisation |
| 🌍 Propositions de pension en amont | Ajoutez ou mettez à jour une ligne dans [REPOSITORY-SOURCES.md](REPOSITORY-SOURCES.md) pour désigner un référentiel pour examen par l'opérateur ; Les propositions de repo-first peuvent désormais utiliser `branch=auto` et `skills_path=auto` |
| 🚫 Modifications manuelles | Les modifications manuelles publiques de `skills_omni/` sont rejetées |
| 🔄 Retraitement | L'amplificateur privé retraite les modifications natives et actualise la base de référence organisée |
