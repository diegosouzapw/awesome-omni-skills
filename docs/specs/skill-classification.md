# Skill Classification and Metadata

This document defines the local classifier used by Omni Skills during validation and build.

## Status

Implemented.

The repository now generates:

- root `metadata.json`
- per-skill `skills/<skill>/metadata.json`

These files are generated by `python3 tools/scripts/validate_skills.py`.

## Purpose

The classifier gives every skill a consistent machine-readable profile before it reaches the catalog.

It currently performs four jobs:

1. parse YAML frontmatter and markdown body
2. normalize category labels to the canonical taxonomy
3. classify maturity and score best practices, quality, and security
4. emit metadata artifacts that build scripts, docs, and CI can consume

## Canonical Taxonomy

The current canonical categories are:

- `development`
- `frontend`
- `backend`
- `fullstack-web`
- `tools`
- `cli-automation`
- `business`
- `product`
- `design`
- `data-ai`
- `ai-agents`
- `machine-learning`
- `devops`
- `testing-security`
- `documentation`
- `content-media`
- `communication`
- `uncategorized`

Legacy or loose labels such as `workflow`, `architecture`, `infrastructure`, `security`, and `testing` are normalized through alias mapping.

## Computed Attributes

### Maturity

Maturity is classified into three levels:

- `L1 metadata` — frontmatter plus minimal body
- `L2 instructions` — substantial written instructions
- `L3 resources` — bundled scripts or richer packaged resources

The classifier also tracks:

- `has_scripts`
- `has_extra_files`

### Best Practices

Best practices are scored from `0` to `100`.

The current heuristic looks at:

- slug quality of `name`
- clarity and length of `description`
- document structure
- examples and code fences
- linked references
- portability signals
- avoidance of hardcoded dates

### Quality

Quality is scored from `0` to `100`.

The current heuristic combines:

- body completeness
- description quality
- metadata completeness
- recency from `date_updated`
- packaged resources
- best practices contribution

Quality is also tiered into:

- `platinum`
- `gold`
- `silver`
- `bronze`
- `starter`

### Security

Security is scored from `0` to `100`.

The current security layer combines:

- static scanning of `SKILL.md` and packaged text files
- script heuristics for risky primitives such as `shell=True`, `pickle.load`, or unsafe archive extraction
- prompt-injection and secret-exfiltration pattern detection
- suspicious path and destructive command detection
- optional ClamAV scanning when `OMNI_SKILLS_ENABLE_CLAMAV=1`
- optional VirusTotal hash lookups when `VT_API_KEY` is configured

Security is emitted as:

- `score`
- `tier`
- `status`
- `findings`
- scanner status for `static`, `clamav`, and `virustotal`

## Generated Metadata Shape

Each skill metadata file contains:

- identity: `id`, `slug`, `display_name`
- taxonomy: `raw_category`, `canonical_category`, `inferred_category`
- authoring metadata: tags, tools, complexity, risk, source, author
- dates and paths
- resource counters
- content signals
- maturity classification
- best practices score
- quality score
- security score and findings
- validation errors, warnings, and status

The root `metadata.json` aggregates:

- repository summary
- taxonomy counts
- skill level distribution
- quality tier distribution
- security tier distribution
- security status counts
- validation status counts
- a compact list of skill summaries

## Workflow Integration

Validation now generates metadata as part of normal repo operations:

```bash
npm run validate
npm run build
npm run taxonomy:report
```

Optional local hook support:

```bash
npm run hooks:install
```

This configures `git` to use `.githooks/pre-commit`, which regenerates metadata and catalog artifacts before commit and stages the generated files.
